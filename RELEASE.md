# Релиз-процесс

Короткий и обязательный чек-лист для каждого деплоя на прод.
Цель — чтобы «локально работало, на проде ломалось» больше не случалось.

## Версионирование (SemVer)

Версия в `package.json` и git tag — `MAJOR.MINOR.PATCH`:

| Что меняем | Bump |
|---|---|
| Только фиксы багов | PATCH (`2.4.0` → `2.4.1`) |
| Новая фича, обратно совместимо | MINOR (`2.4.1` → `2.5.0`) |
| Сломали API / схема несовместима | MAJOR (`2.5.0` → `3.0.0`) |

## Шаги релиза (5 минут)

### 1. Перед коммитом — определиться с типом версии

Смотри `git log <последний-tag>..HEAD` — какие там префиксы:
- только `fix(...)` → PATCH
- хотя бы один `feat(...)` → MINOR
- сломали миграцию / API / удалили endpoint → MAJOR

### 2. Обновить `package.json`

```bash
# либо вручную поправить "version"
# либо через npm:
npm version patch    # 2.4.0 → 2.4.1
npm version minor    # 2.4.1 → 2.5.0
npm version major    # 2.5.0 → 3.0.0
```

`npm version` сам создаст коммит `chore(release): X.Y.Z` и git tag `vX.Y.Z`.

**Если правишь вручную** — после правки:
```bash
git add package.json package-lock.json
git commit -m "chore(release): 2.5.0"
git tag v2.5.0
```

### 3. Заполнить `CHANGELOG.md`

Перенести записи из секции `[Unreleased]` в новую секцию `[X.Y.Z] — YYYY-MM-DD`. Перечислить **Added / Changed / Fixed / Removed / Security / Deploy**.

**Секция Deploy** — самое важное. Сюда писать всё, что **админу прода нужно знать сверх обычного `git pull && rebuild`**:
- новые env-переменные
- неочевидные миграции
- зависимости от внешних сервисов
- известные ограничения (например «не забыть освободить диск»)

### 4. Push + tag

```bash
git push origin main
git push origin v2.5.0
```

### 5. Деплой на прод

См. `deploy-2026-05-17.md` (универсальный шаблон). Главное:
- бэкап БД
- `df -h /` — нужно ≥ 8 GB
- билд **по одному сервису**, `docker builder prune` между ними
- smoke-чеклист

### 6. После деплоя

В `CHANGELOG.md` секция `[X.Y.Z]` уже стоит как «released». Создать на GitHub Release из тега (опц.) — туда вставить тот же текст.

---

## Что попадает в `[Unreleased]`

После каждого коммита, который **меняет поведение пользователя или админа**, добавляй строку в `[Unreleased]` сразу. Не откладывай — забудешь.

Не нужно для:
- рефакторинга без изменения поведения
- доков
- chore-коммитов (`chore: refresh GitNexus index stats`)

## Откат версии

Если новая версия сломала прод:
```bash
ssh user1@87.242.102.125
cd /opt/pilingtrack
git fetch --tags
git reset --hard v<PREVIOUS>     # например v2.3.0
docker builder prune -af
docker compose build app workers
docker compose up -d app workers
```

⚠️ **Миграции БД** обычно additive (только ADD COLUMN / CREATE TABLE), откатывать не нужно. Если миграция деструктивная — об этом должна быть запись в `CHANGELOG.md` секция **Deploy** с инструкцией.

## Локальная копия прод-БД (защита от «локально работает, на проде ломается»)

Большинство сегодняшних факапов проявлялись на проде из-за объёма данных, дубликатов, реальных user-сценариев — на чистой dev-БД их не видно. Решение: держать локально **снимок прод-БД** и тестировать критичные изменения на нём.

### Команды

```bash
npm run db:status              # какая БД сейчас активна
npm run db:use-prod            # переключить на снимок прод-БД
npm run db:use-dev             # вернуть тестовую БД
npm run db:refresh-prod-snapshot  # обновить снимок (раз в неделю)
```

После переключения **перезапусти `npm run dev`** — Next.js кэширует env при старте.

### Что внутри

- БД `pilingtrack_prod_copy` (рядом с `pilingtrack_test` в локальном Postgres)
- 34 отчёта, 10 юзеров, 8 установок — точная копия прода на момент последнего snapshot
- Пароли пользователей не меняются — можно логиниться теми же креденшелами, что и на проде

### Когда что использовать

| Задача | Какая БД |
|---|---|
| Эксперимент со схемой / новая миграция / тестовые данные | `db:use-dev` |
| Воспроизвести баг с прода / проверить performance на реальном объёме / финальная проверка перед релизом | `db:use-prod` |
| Тест RLS / multi-tenant логики | `db:use-prod` (там настоящие tenantId) |

### ⚠️ Что нельзя

- **Не пушить миграции, протестированные только на dev-БД**, если изменения затрагивают данные. Сначала `db:use-prod` → проверь на снимке.
- **Не делать `prisma db push` против `pilingtrack_prod_copy`** — это может расходиться со схемой и потом удивит.
- **Не использовать `db:use-prod` для seed / drop-recreate flow** — снимок одноразовый, потом перекачивать.

---

## Чем это лучше чем «просто push в main»

| Без процесса | С процессом |
|---|---|
| Прод-версия = «случайный коммит» | `git tag` фиксирует точку, к которой откатиться |
| Что в релизе — никто не помнит | `CHANGELOG.md` за 30 секунд расскажет |
| Деплой ломает прод — никто не знает почему | секция **Deploy** в `CHANGELOG.md` предупреждает |
| Откат — гадание на git log | `git reset --hard v2.3.0` — одна команда |

5 минут на релиз вместо часа разбора полётов.
