# PilingTrack Change Control

How changes are classified, gated, and reviewed in this repo ‚Äî and the
non-negotiable rules, each with the incident that created it. Audience:
an engineer or AI session with zero prior context.

**Prime directive:** every rule below exists because its violation already
happened here and cost something. Do not relax a rule because it "seems
overly cautious" ‚Äî check the incident first (one-line citations below;
full chronicle in the `pilingtrack-failure-archaeology` skill).

---

## 1. Classify your change first

| Your change touches‚Ä¶ | Class | Gate (see ¬ß2) |
|---|---|---|
| Ordinary app/UI/module code | **Routine** | Fast loop + `npm run verify` + `qa-checklist` skill |
| `prisma/schema.prisma` or `prisma/migrations/**` | **DB schema** | `create-migration` skill, ALL of its steps |
| `src/services/auth/**`, `src/core/security/**`, `src/lib/rate-limiter.ts`, tenant/media access | **Security-critical** | Test-first; never refactor without tests |
| Anything shipping to orionpiling.ru | **Prod deploy** | Human-driven, step-by-step; `deploy` skill; NEVER autonomous |
| Deleting code that "looks unused" | **Suspect deletion** | Read ¬ß3.2 (dormant code) before deleting anything |
| Visual restyle not explicitly requested | **Forbidden** | Don't. See ¬ß3.3 |

A change can be in several classes at once (e.g. a new tenant-scoped table =
DB schema + security-critical). Apply every matching gate.

When unsure which class applies, the safe default is the stricter one.

## 2. Gates per class

### 2.1 Routine code

- Fast inner loop while coding: `npm run lint` + `npm run test:unit` (seconds).
- Before a commit that will be pushed / PR'd / deployed: **`npm run verify`**
  ‚Äî the single canonical gate. It runs, in order:
  `db:check-migrations ‚Üí lint ‚Üí typecheck ‚Üí test:unit ‚Üí build ‚Üí test:smoke:auth-access`
  (verified in `package.json` `scripts.verify`, 2026-07-07).
- Full checklist, including change-specific checks `verify` cannot judge
  (API wrappers, tenant queries, logging, Russian labels): the
  **`qa-checklist`** skill. Do not duplicate it here ‚Äî go read it.
- Diff must be **surgical**: every changed line traces to the request
  (CLAUDE.md ¬ß3). Reviewers reject drive-by "improvements".

### 2.2 DB schema / migrations

Owned end-to-end by the **`create-migration`** skill. The rules it enforces,
so you know why you're being sent there:

- **One migration = one logical change.** Bundling 5+ models into one
  migration is a listed pitfall in CLAUDE.md ‚Äî splits keep review tractable.
- **Read the generated SQL yourself.** Prisma auto-generates `DROP COLUMN`
  for renames it can't see. Incident: 2026-05-30, an autogenerated diff
  silently dropped a column ‚Äî that incident is why the
  `npm run db:check-migrations` destructive-statement guard exists
  (`scripts/check-migrations.js` + allowlist
  `scripts/.migration-guard-baseline.txt`).
- **New tenant-scoped table ‚Üí RLS policy in a separate `_rls` migration.**
  Pattern and template live in `create-migration`.
- **The stale-migrate-image deploy gotcha:** the `migrate` Docker service
  bakes `prisma/migrations` into its image at build time ‚Äî deploying
  `app`+`workers` without rebuilding `migrate` makes it silently skip your
  migration. Rule: any diff adding a `prisma/migrations/*` folder means the
  deploy must `docker compose build migrate app workers` and then verify in
  the DB that the migration actually landed. Full incident and mechanism:
  `pilingtrack-failure-archaeology` C1. Commands:
  `create-migration` ("Deploy reminder") and
  `docs/runbooks/008-manual-deploy.md` ("Migrations" section).

### 2.3 Security-critical files

Files (from CLAUDE.md, verified present 2026-07-07):

- `src/services/auth/` ‚Äî authentication, session, token logic
  (incl. `authorization-service.ts`, `resource-access-service.ts`)
- `src/core/security/` ‚Äî `encryption.ts`, `refresh-tokens.ts`,
  `tenant-enforcement.ts`, `idempotency.ts`
- `src/lib/rate-limiter.ts`

Gate: **test-first, always.** Write a test that reproduces the current
behavior, then change the code. **Never refactor these "cleanly" without
tests** ‚Äî even a refactor that "obviously preserves behavior". ¬ß3.1 below is
the incident proving that an experienced session's "obviously equivalent"
tenant-guard rewrite was wrong twice in one evening.

This gate coexists with the keep-tests-lean rule (¬ß3.4): lean means *few,
essential* tests ‚Äî the one guard test for a destructive/auth operation is
exactly the kind that must exist.

### 2.4 Prod deploys

- **The human operator drives every prod deploy, step-by-step.** Sessions
  prepare commands and pre-flight checks; the operator (or a session under
  the operator's live direction over SSH) executes them one at a time and
  reads each output before the next. Never deploy autonomously, never chain
  the whole runbook into one unattended script.
- Use the **`deploy`** skill ‚Äî it generates the paste-ready command block and
  auto-detects whether `migrate` must be rebuilt (¬ß2.2 gotcha).
- Canonical runbook: `docs/runbooks/008-manual-deploy.md` (zero-downtime:
  build first, swap when healthy; build `app` and `workers` **sequentially**
  ‚Äî a parallel build filled the 30 GB VPS disk to 100%).
- Pre-flight data checks for risky migrations run as **separate statements**,
  not one multi-statement paste (lesson from the 2026-06-30 deploy, where a
  unique-constraint migration failed on duplicate prod data mid-deploy).

## 3. Non-negotiables (each with its incident)

### 3.1 Tenant-scoped queries fail closed ‚Äî never `IS NULL OR tenantId`

**Rule:** a tenant-scoped query with a missing `tenantId` must **throw**
(fail closed), never fall back to `IS NULL OR "tenantId" = $x`. With a null
tenant, the `IS NULL OR` form returns **every tenant's rows** ‚Äî an IDOR
(Insecure Direct Object Reference: reading data you're not authorized for).

**Incident (2026-05-31):** the pattern was found live, fixed, then
*re-introduced within the same evening*: commit `22bf1a5`
(`revert(analytics): restore IS NULL OR tenant guard on Equipment select`,
19:49) restored the dangerous guard on the plausible-sounding reasoning that
strict `= NULL` would make lists "silently empty". Six minutes later,
`10fc759` (19:55, `fix(analytics): fail closed on missing tenantId`)
replaced both branches with a throw + a regression test. Lesson: the
antipattern *looks like* defensive coding; that is exactly why it keeps
coming back. If you see `IS NULL OR` in a tenant filter, it is a bug, not a
feature.

**Policy home:** `src/services/auth/resource-access-service.ts` ‚Äî multi-tenant
mode fails closed on a user without `tenantId` (verified 2026-07-07).

**Do not confuse** with the DB-level RLS policies, whose
`current_setting('app.current_tenant', true) IS NULL OR ‚Ä¶` branch is the
deliberate DB *audit-mode* allowance (GUC unset), not the app-layer
antipattern ‚Äî see `create-migration` for that distinction.

**Known open edge (deferred by owner, 2026-06-28):** `ensureTenantAccess`
in `resource-access-service.ts` unconditionally bypasses ADMIN/DISPATCHER
roles, and the whole check is a no-op while `MULTI_TENANT_MODE` is unset in
prod. Whether that bypass is intentional platform-admin design or a gap is
an **explicitly deferred decision** ‚Äî do not silently "fix" it either way;
re-surface only when second-tenant onboarding starts (decision horizon
2026-11-24).

### 3.2 Dormant code must NOT be deleted

**Rule (explicit owner decision):** code that is currently unreachable or
unused may still be *dormant, not dead* ‚Äî it activates later. Before
deleting anything "unused", check whether it is on this list or looks like
it belongs on it:

- **Telemetry ingestion/read path** (`TelemetryRecord` model and its
  services) ‚Äî dormant until a hardware telemetry box is connected to a rig;
  it was deliberately kept during the 2026-05-31 simplify-and-harden pass
  and has since received live security fixes (`370e592`, 2026-07-01), which
  is proof it is maintained, not abandoned.
- **Multi-tenancy code** (tenancy services, `MULTI_TENANT_MODE` paths, RLS
  machinery) ‚Äî single tenant `orion` today, but the hybrid-SaaS decision is
  due **2026-11-24** and the owner's direction is explicit: do NOT remove
  tenancy code.
- **`ReportPhoto` model** (`prisma/schema.prisma`, present as of
  2026-07-07) ‚Äî a dormant evidence path; the live photo path is `Media`.
  Do not delete it and do not *build on* it either ‚Äî see the
  `report-evidence-model` skill.

If you find genuinely dead code that is not on this list: **mention it,
don't delete it** (CLAUDE.md ¬ß3), unless your own change orphaned it.

### 3.3 No unsolicited UI redesigns

**Rule:** visual/UX restyling happens only when explicitly requested. Do not
"improve" typography, navigation grouping, color tokens, dark mode, KPI
layouts, or page headers as a side effect of another task.

**Incident (2026-04-30):** an entire 5-commit redesign wave ‚Äî hero KPIs,
typography discipline, dark-mode toggle, grouped sidebar nav, wider
content ‚Äî was fully reverted the same day: `b490dcc`, `38890c3`, `213be3f`,
`13febdf`, `afb3712` (all `Revert "‚Ä¶"` commits, verified in git log). The
work was competent and completely unwanted. Net result: two waves of diff
churn and zero product value.

### 3.4 Keep tests lean

**Rule (owner preference, standing since ~2026-06):** no sprawling new test
files. The owner reads diffs directly and perceives test proliferation as
clutter. Default to adding the few necessary assertions to the module's
*existing* test file (`src/**/__tests__/`). Write only **essential safety
guards** ‚Äî above all, the single test that protects a destructive or
auth-sensitive operation (e.g. the eligibility guard on hard-deleting a
site). Skip coverage-filler.

This does not override ¬ß2.3: security-critical changes still get their
test *first* ‚Äî it caps quantity, not the discipline.

If omitting tests shifts data-loss risk onto the operator, say so plainly
instead of silently complying.

### 3.5 GitNexus gates (from CLAUDE.md ‚Äî mandatory, not advisory)

The repo is indexed by GitNexus (code knowledge graph). CLAUDE.md makes two
of its tools blocking gates:

- **Before editing any function/class/method:**
  `impact({target: "symbolName", direction: "upstream"})` ‚Äî report blast
  radius; HIGH/CRITICAL risk must be warned about before proceeding.
- **Before committing:** `detect_changes()` ‚Äî verify only expected symbols
  and flows are affected.
- Never rename symbols by find-and-replace ‚Äî use the GitNexus `rename` tool.

Index stale? `node .gitnexus/run.cjs analyze` from the repo root
(`.gitnexus/run.cjs` verified present 2026-07-07). Tool details:
`.claude/skills/gitnexus/*/SKILL.md`.

### 3.6 `docs/audit.md` is a snapshot ‚Äî never trust "open" without verifying

**Rule:** `docs/audit.md` (and any roadmap/status document or memory) is a
point-in-time snapshot, not a live backlog. Its own header says so, in
bold, in Russian: verify each item against current code before acting.

**Incident (May 2026):** ~30% of items marked "open" were already closed ‚Äî
acting on the document at face value would have duplicated finished work.
Same failure mode recurred with roadmap status on 2026-06-24 (two "pending"
phases were fully built and merged).

**How to apply:** before treating any "open" item as actionable ‚Äî grep for
the symptom, open the cited file, `git log --grep '(C-1)'`-style search for
closure commits. Already closed ‚Üí update the doc, don't redo the work. New
finding ‚Üí new tag (`N-<next>`), never revive an old number. Audit older
than ~2 months ‚Üí propose a refresh instead of working from it pointwise.

### 3.7 Fix findings in touched code now ‚Äî "pre-existing / not mine" is invalid

**Rule (owner decision):** this codebase is entirely Claude-authored across
sessions. When a review or your own work surfaces a **security or clear
correctness** bug in code you are touching or immediately adjacent to it,
fix it in this change ‚Äî do not park it as "pre-existing, out of scope".
Judgment still applies to scope (don't balloon a task into unrelated
refactors), and ¬ß3.2's don't-delete-dormant-code rule still holds: that
rule is about not gratuitously rewriting working code, not a license to
ignore defects.

## 4. Behavioral baseline (reference, don't restate)

CLAUDE.md ¬ß¬ß1‚Äì4 are the load-bearing behavioral rules for every change and
this skill assumes them rather than repeating them:

1. **Think before coding** ‚Äî state assumptions; multiple interpretations ‚Üí
   present them, don't pick silently.
2. **Simplicity first** ‚Äî minimum code; no speculative flexibility.
3. **Surgical changes** ‚Äî every changed line traces to the request.
4. **Goal-driven execution** ‚Äî turn the task into a verifiable check, loop
   until it passes.

Also from CLAUDE.md, worth internalizing as review criteria: API routes
always wrapped in `withApi`/`withMutation` (never inline CSRF/rate-limit),
`safeParse` validation, `logger.*` not `console.log`, files under ~500
lines, new domain code imported via `@/modules/<x>` facades.

## 5. Review expectations (what gets a change rejected)

- Diff contains lines that don't trace to the request (drive-by cleanups,
  reformatting, "improved" adjacent code).
- A tenant filter using `IS NULL OR` (¬ß3.1) ‚Äî instant reject.
- Security-critical file changed with no test written first (¬ß2.3).
- Migration bundling several unrelated schema changes, or an unreviewed
  `DROP`/`TRUNCATE` (¬ß2.2).
- New test files where existing files would do, or coverage-filler (¬ß3.4).
- Unrequested visual redesign (¬ß3.3).
- Deleted dormant code (¬ß3.2).
- Commit made without `detect_changes()` / edit made without `impact` (¬ß3.5).
