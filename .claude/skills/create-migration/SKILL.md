# Create a Prisma migration (safely)

A guided wrapper around `prisma migrate dev` that bakes in this project's
migration rules. Prompted by the 2026-05-30 incident where an autogenerated
diff silently dropped a column.

## Before generating

1. Confirm the schema change in `prisma/schema.prisma` is **one logical change**.
   If it touches 5+ models, stop and ask the user to split it — one migration =
   one logical change (CLAUDE.md rule, keeps PR review tractable).

2. Ask the user for a short, descriptive migration name (snake_case), e.g.
   `add_report_archived_flag`.

## Generate

Run against the LOCAL dev DB only (never prod):

```bash
npx prisma migrate dev --name <name>
```

This creates `prisma/migrations/<timestamp>_<name>/migration.sql` and applies
it locally.

## Mandatory review — read the generated SQL

Open the new `migration.sql` and read it in full. Specifically check for
statements Prisma may have auto-generated that you did NOT intend:

- `DROP COLUMN`, `DROP TABLE`, `TRUNCATE` → **data loss**. If unintended, fix
  the schema and regenerate. If intended, see "Allowlisting" below.
- A rename auto-detected as drop+add (Prisma can't see renames) → data loss.
  Use `@map` or a manual SQL migration instead.

Then run the guard exactly as CI will:

```bash
npm run db:check-migrations
```

- **Passes** → no un-reviewed destructive statements. Good.
- **Fails** → the guard found a DROP/TRUNCATE. Do NOT allowlist reflexively.
  Confirm it's intentional first. To allowlist, add the migration's directory
  name to `scripts/.migration-guard-baseline.txt` with a comment explaining
  why the destructive change is safe (see existing entries for the format).

## Destructive migrations on prod

If the migration drops a column/table and prod has data in it, check prod
first before deploying:

```bash
# on the VPS
docker compose exec postgres psql -U piling -d pilingtrack \
  -c 'SELECT count(*) FROM "<Table>";'
```

## Deploy reminder (don't forget the migrate service)

A new migration folder means the prod deploy MUST rebuild the `migrate`
service — it bakes `prisma/migrations` into its image at build time:

```bash
docker compose build migrate app workers   # NOT just `app workers`
docker compose up -d app workers
# verify it landed (don't trust exit 0):
docker compose exec postgres psql -U piling -d pilingtrack \
  -c "SELECT migration_name FROM _prisma_migrations ORDER BY finished_at DESC NULLS LAST LIMIT 1;"
```

Full deploy flow: the `/deploy` skill (it auto-detects the new migration).
