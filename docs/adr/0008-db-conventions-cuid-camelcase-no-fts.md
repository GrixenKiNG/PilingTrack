## Context

Чек-листы качества Postgres, применяемые в проекте, расходятся с фактической схемой по трём фундаментальным пунктам:

1. **Первичные ключи** — правила рекомендуют `BIGINT GENERATED ALWAYS AS IDENTITY`; в схеме все **53 модели** используют `String @id @default(cuid())`.
2. **Имена идентификаторов** — правила рекомендуют `snake_case`; схема использует Prisma-дефолт: таблицы `PascalCase`, колонки `camelCase` (в кавычках), **ни одного `@map`/`@@map`** — то есть SQL-имена совпадают с именами в коде.
3. **Полнотекстовый поиск** — правила упоминают FTS (`tsvector`/GIN); в базе его нет.

Эти решения зафиксированы в **31 миграции**, во всех RLS-политиках, raw-SQL запросах (`$queryRaw`), индексах и скриптах. Каждый повторный аудит тратит время на «переоткрытие» этих пунктов.

## Decision

Все три пункта — **осознанно принятые решения, а не долг**. Оставляем как есть; из аудиторских чек-листов эти пункты закрываются ссылкой на данный ADR.

1. **cuid остаётся.** ID генерируются приложением (работает при офлайн-создании и в воркерах без round-trip к БД), не перечислимы снаружи (нет IDOR через `id+1`), уникальны глобально (безопасный merge данных двух тенантов). Штраф — ~2× размер индекса против BIGINT — при масштабе проекта (сотни строк в день, дамп базы 196 КБ) неизмерим.
2. **camelCase остаётся.** Единственное преимущество snake_case — привычность DBA-инструментам и отсутствие кавычек в ручном SQL. Цена перехода — переименование каждой таблицы/колонки + правка всех RLS-политик, партиционирования, raw-SQL, индексов и скриптов с нулевой пользой для пользователя. Совпадение имён SQL ↔ TypeScript — само по себе ценность (меньше слоёв трансляции).
3. **FTS не вводим.** В приложении нет ни одного запроса поиска по подстроке (`contains` не используется нигде в `src/`, вся фильтрация — точные совпадения или клиентская). Строить FTS-инфраструктуру без потребителя — спекулятивная функциональность (CLAUDE.md §2).

## Consequences

### Positive
- Аудиты перестают поднимать закрытые вопросы (ссылка на ADR вместо повторного разбора).
- Ни одной строки кода/миграций менять не нужно.

### Negative
- Ручной SQL в psql требует кавычек: `SELECT "tenantId" FROM "Report"`.
- Индексы по text-PK крупнее, чем по BIGINT (на текущем масштабе неизмеримо).

### Risks
- Если появится реальный поиск (например, по названиям свай/объектов в формах) — сначала `pg_trgm` + GIN по конкретной колонке, а не полноценный tsvector-конвейер.

## Alternatives Considered

1. **Миграция PK на BIGINT identity**
   - Pros: компактные индексы, «канонично» для Postgres.
   - Cons: перешивание FK во всех 53 моделях, потеря офлайн-генерации ID, появление перечислимости, недели работы.
   - Why not chosen: цена катастрофична, выгода на этом масштабе нулевая.

2. **Переименование в snake_case через `@map`**
   - Pros: SQL без кавычек.
   - Cons: 31 миграция истории, все RLS-политики и raw-SQL переписать; двойная система имён (code ≠ SQL).
   - Why not chosen: чистая косметика с большим blast radius.

3. **Ввести FTS «на вырост»**
   - Pros: готовность к будущему поиску.
   - Cons: нет ни одного потребителя; триггеры/индексы, которые никто не вызывает.
   - Why not chosen: правило Simplicity First — спекулятивная функциональность запрещена.

## Implementation Notes

Ничего не внедряется. Пересмотр триггерится одним из событий:
- второй **реальный** тенант + измеримые проблемы с размером индексов (маловероятно);
- продуктовая задача «поиск по текстовым полям» (тогда — точечный `pg_trgm`);
- шардирование/партиционирование, где BIGINT-ключи дали бы измеримый выигрыш.
