## 🔴 CRITICAL (5 находок)

### C1. Seed-скрипт импортирует не тот Prisma Client
**Файл:** `prisma/seed.ts:1`
**Проблема:**
```typescript
import { PrismaClient } from '@prisma/client'; // ← DEFAULT client
```
Schema генерирует клиент в `../src/generated/postgres-client`, но seed импортирует дефолтный `@prisma/client`, который может указывать на другой (или несуществующий) datasource.
**Воздействие:** Seed-данные могут не попасть в целевую БД; в worst case — seed падает без объяснения.
**Рекомендация:** `import { PrismaClient } from '../src/generated/postgres-client';`
**Сложность:** 1 строка

---

### C2. Access token (JWT) без jti — нет механизма отзыва
**Файл:** `src/services/auth/session-service.ts:55-65`
**Проблема:** `createSessionToken()` не включает `jti` (JWT ID) в payload. Украденный access token действует полные 12 часов без возможности отзыва.
**Контекст:** Refresh tokens правильно реализуют rotation + family revocation (`src/core/security/refresh-tokens.ts`). Но access tokens — нет.
**Воздействие:** При компрометации сессии атакующий имеет 12 часов неотзываемого доступа.
**Рекомендация:**
1. Добавить `jti: crypto.randomUUID()` в JWT payload
2. Вести Redis set `revoked:jti` с TTL = SESSION_TTL
3. В `requireAuth()` проверять `!isRevoked(jti)`
**Сложность:** ~50 строк

---

### C3. Нет валидации SESSION_SECRET при старте приложения
**Файл:** `src/instrumentation.ts`
**Проблема:** `register()` не валидирует наличие и длину `SESSION_SECRET`. Если переменная не задана — dev-fallback (`dev-only-session-secret-change-me`) используется **молча** и в `session-service.ts` просто пишется `console.warn`. В production это означает все сессии подписаны известным ключом.
**Воздействие:** Полная компрометация аутентификации.
**Рекомендация:** В `register()` добавить:
```typescript
if (process.env.NODE_ENV === 'production' && !process.env.SESSION_SECRET) {
  throw new Error('SESSION_SECRET is required in production');
}
```
**Сложность:** 5 строк

## 🟠 HIGH (14 находок)

### H1. X-Frame-Options конфликт: SAMEORIGIN vs DENY
**Файлы:** `next.config.ts:24` + `src/middleware.ts:60`
- `next.config.ts` → `SAMEORIGIN` (разрешает embedding с того же домена)
- `middleware.ts` → `DENY` (запрещает embedding полностью)
Middleware перезаписывает next.config для API routes (`/api/*`), но для page routes — `SAMEORIGIN`.
**Рекомендация:** Унифицировать в `DENY` (приложению не нужен embedding).

### H2. Dead code: `src/core/api-wrapper/index.ts` (229 строк)
**Проблема:** При `"moduleResolution": "bundler"`, файл `api-wrapper.ts` имеет приоритет над `api-wrapper/index.ts`. Все 30 route-файлов резолвят в `api-wrapper.ts`. Версия в `api-wrapper/index.ts` с SLO, bulkhead, cache — мёртвый код.
**Рекомендация:** Удалить `src/core/api-wrapper/index.ts` или объединить логику.

### H9. ENCRYPTION_KEY не валидируется
**Файл:** `.env.example` содержит `ENCRYPTION_KEY=""`. Если не задан, `src/core/security/encryption.ts` (если есть) может fallback на небезопасное значение.

### H10. WebSocket server без rate limiting
**Файл:** `src/realtime/server/ws-server.ts`
Нет ограничения на частоту подключений или сообщений. DoS-атака через WebSocket.

### H11. Redis fallback на in-memory — без предупреждения в health-check
Rate limiter и cache деградируют на in-memory Map, но `/api/health` не показывает Redis как отдельный check.

### H12. `bcryptjs` вместо native `bcrypt`
`bcryptjs` — чистый JavaScript, ~3x медленнее native `bcrypt`. На Node.js 22 + Alpine лучше использовать native.

### H13. Soak test timeout 120 мин — но k6 soak script может быть длиннее
Мы снизили с 480 до 120 мин, но нужно проверить, что soak.test.js укладывается в 120 мин.

### H14. PIN lookup: legacy users сканируются O(n) при каждом входе
**Файл:** `src/services/auth/auth-service.ts:216-245`
Пользователи без `pinLookup` проходят через full table scan. Opportunistic upgrade может fail silently.
**Рекомендация:** Одноразовая миграция для pre-fill всех `pinLookup`.

## 🟡 MEDIUM (16 находок)

### M1. 90 production-зависимостей — раздутый bundle
Для Next.js проекта 90 prod deps — много. Потенциально неиспользуемые: `mqtt`, `@bull-board/*`, `kafkajs`, `nats` (если не деплоится Kafka/NATS).

### M2. `src/lib/` — 46 файлов в одной директории
Нужна группировка: `lib/auth/`, `lib/cache/`, `lib/db/`, `lib/validation/`.

### M3. 10 TODO/FIXME в коде
```
grep -rn 'TODO\|FIXME' src → 10 matches
```
Каждый — потенциальный незакрытый дефект.

### M4. `src/core/event-bus/kafka-nats-adapters.ts` (436 строк) — мёртвый код?
Kafka и NATS adapters реализованы, но нет конфигурации для их использования в production. Если in-memory + Redis event bus достаточен — удалить.

### M5. CSP без `report-uri`
`Content-Security-Policy` в `next.config.ts` не содержит `report-uri` или `report-to`. Нарушения CSP не фиксируются.

### M6. CORS `*.domain` wildcard — subdomain takeover risk
`isOriginAllowed()` поддерживает `*.pilingtrack.ru` — если субдомен скомпрометирован, он получает полный CORS доступ.

### M7. Нет `images` config в `next.config.ts`
Next.js `images.remotePatterns` не настроен — `<Image>` не может загружать внешние изображения.

### M8. Helm `imagePullSecrets` не настроен
`infra/helm/pilingtrack/` — если GHCR registry приватный, pods не смогут скачать образы.

### M9. Helm `readOnlyRootFilesystem: false`
Security context не включает read-only root FS — контейнер может записывать в свою FS.

### M10. Monitoring exporters (Prometheus targets) указывают на несуществующие сервисы
Prometheus конфигурация может ожидать exporters, которые не деплоятся.

### M11. `revalidateInBackground` без backpressure
Фоновая ревалидация кэша может создать thundering herd при массовом expired cache.

### M12. Нет индекса на `SitePilePlan.pileGradeId`
JOIN через `pileGradeId` без индекса — slow query при большом количестве plan записей.

### M13. `registeredDevices` в telemetry хранятся in-memory
`src/app/api/telemetry/ingest/route.ts` — устройства верифицируются из памяти. При рестарте — все устройства перезапрашиваются.

### M14. PDF worker — нет retry logic при S3 upload failure
BullMQ retry работает на уровне job, но внутри job S3 upload не обёрнут в retry.

### M15. `Date.now()` как cursor в sync endpoints
`src/app/api/sync/route.ts:108` и `sync/batch/route.ts:100` — `cursor: Date.now()` не монотонный (NTP drift, clock skew). Для sync cursor лучше использовать DB sequence.

### M16. Docker builder stage содержит исходники
`Dockerfile` копирует весь `src/` в builder stage. При публикации builder image (не финальной) — утечка исходного кода.

## ✅ ЧТО СДЕЛАНО ХОРОШО

### Архитектура (8.5/10)
- **DDD + CQRS** — чёткое разделение commands/queries/domain/infrastructure
- **Transactional Outbox** — гарантированная доставка событий
- **Aggregate Pattern** — инварианты защищены на уровне домена
- **Circuit Breakers** — graceful degradation при недоступности DB
- **Leader Election** — корректная работа workers в multi-instance
- **Dead Letter Queue** — события не теряются

### Безопасность (7.5/10)
- **4-уровневая CSRF защита** (Sec-Fetch-Site, Origin, Referer, baseline)
- **RBAC с 23 abilities** — гранулярный контроль доступа
- **bcrypt + HMAC PIN auth** — O(1) lookup с защитой от brute-force
- **Refresh Token Rotation** — family-based revocation при компрометации
- **Tenant isolation** — fail-closed `ensureTenantAccess()`
- **Constant-time PIN comparison** (исправлено в этой сессии)
- **Rate limiting** — Redis-backed с in-memory fallback

### Операции (7.5/10)
- **Helm chart** — production-ready с HPA, PDB, network policies
- **CI/CD** — 7-stage pipeline (lint → test → build → e2e → security → docker → deploy)
- **Health checks** — Kubernetes-compatible (liveness, readiness, health)
- **SLO monitoring** — burn rate alerts
- **ArgoCD** — GitOps deployment

### Тестирование (6.5/10)
- **607 тест-кейсов** — хорошая baseline
- **Domain aggregate tests** — бизнес-логика покрыта
- **E2E tests** (Playwright) — golden path проверен
- **Worker tests** — outbox, projection, pdf workers покрыты
