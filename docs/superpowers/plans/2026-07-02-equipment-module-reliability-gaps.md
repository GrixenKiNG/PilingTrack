# План: пробелы достоверности модуля установок (пункты 2, 4, 5)

Дата: 2026-07-02
Статус: proposed (на обсуждение)
Источник: браузерный аудит модуля установок под ADMIN + OPERATOR (2026-07-02).
Пункты 1 и 3 из того же аудита уже исправлены (коммит на chore/project-skills).

## Контекст

Модуль установок архитектурно целостный и сквозно связный: данные, введённые в
отчёте/ТО, корректно всплывают на дашборде, в мониторинге и в аналитике надёжности.
Проблема не в проводке, а в том, что три ключевые метрики опираются на данные,
которые в реальной эксплуатации Ориона **никто не вводит вручную**. Ниже — три
пробела и варианты решения. Это НЕ баги проводки, а продуктовые решения о том,
"откуда берётся правда".

## Пункт 2 — «Готовность парка 100%» при живом открытом ремонте

**Симптом:** вкладка «Надёжность ТО» показывает «Готовность парка 100.0% (доля
времени без ремонтов)», хотя на установке висит незакрытый ремонт. Метрика считает
только *завершённые* ремонты с проставленными `startedAt → completedAt`. Установка
может простоять неделю в незакрытом ремонте, а готовность покажет 100%.

**Причина:** availability = 1 − (суммарная длительность закрытых ремонтов / период).
Открытый ремонт даёт нулевую длительность до момента закрытия с обоими таймстемпами.

**Варианты:**
- **A (рекомендуется).** Считать простой открытого ремонта «на лету»: для записей
  без `completedAt` брать интервал `startedAt → now()` (или `createdAt → now()`,
  если `startedAt` пуст). Тогда готовность падает в реальном времени, пока ремонт
  открыт. Малое изменение в `computeFleetKpi` (src/lib/fleet-kpi.ts).
- **B.** Оставить формулу, но приписать метрике честную оговорку в UI: «по закрытым
  ремонтам» — и вынести отдельный счётчик «сейчас в открытом ремонте: N».
- **C.** Ничего не менять до появления дисциплины ввода таймстемпов (наименее ценно).

**Критерий готовности (для A):** unit-тест `computeFleetKpi` с одним открытым
ремонтом (`completedAt=null`, `startedAt` = 2 дня назад) → availability < 100%.

**Оценка:** A — ~30 строк + тест, полдня.

## Пункт 4 — Моточасы пусты у 7 из 8 установок → планирование ТО по наработке мертво

**Симптом:** в списке парка только Banut показывает «5 670 ч», у остальных «— ч».
`engineHoursTotal` / `nextMaintenanceAtHours` не заполнены. При этом весь механизм
`MaintenancePlan` с триггером `HOURS` и `pm-scheduler` построен вокруг этих полей —
без наработки ТО-по-моточасам не срабатывает вообще.

**Причина:** нет ни телеметрии (бокс не установлен — честно задокументировано), ни
привычки вносить моточасы вручную через журнал наработки (`MeterReading`).

**Варианты:**
- **A (рекомендуется, дёшево).** Сделать ввод моточасов частью привычного потока:
  добавить необязательное поле «моточасы на конец смены» в форму отчёта оператора.
  Раз в смену оператор и так открывает форму — одно число закрывает данные по всему
  парку. `MeterReading` уже существует; нужен маппинг «отчёт → показание».
- **B.** Оставить только ручной журнал наработки в модуле ТО (уже есть), но
  честно пометить план ТО-по-моточасам как «неактивен без наработки», чтобы не
  создавать ложного впечатления работающего планирования.
- **C.** Ждать телеметрии (Teltonika/Galileosky) — самый полный, но не в этом году.

**Критерий готовности (для A):** отправка отчёта с полем моточасов создаёт
`MeterReading`; `evaluatePlanDue` для HOURS-плана начинает возвращать `due_soon/overdue`.

**Оценка:** A — форма + маппинг + миграция связи, 1–2 дня. Требует продуктового
решения (не перегрузить форму оператора).

**Открытый вопрос для владельца:** готовы ли операторы вносить моточасы раз в смену?
Если нет — пункт 4 закрывается только телеметрией, и тогда честнее вариант B.

## Пункт 5 — «Зависшие» наряды и осмотры копятся без просрочки и эскалации

**Симптом:** на карточке установки — наряды ТО «В работе» и осмотры «В процессе»,
старейшие от 10.06, ни один не закрыт. Список растёт и перестаёт быть сигналом.
Это же тянет вниз «Выполнение ППР» (24%, 4 из 17) — знаменатель раздут зомби-нарядами.

**Причина:** нет ни срока на наряд/осмотр, ни авто-пометки просрочки, ни эскалации.
Открытая запись живёт вечно в одном и том же статусе.

**Варианты:**
- **A (рекомендуется).** Ввести «возраст в статусе»: наряд/осмотр, открытый дольше
  N дней (напр. 14) без движения, помечается «просрочен» (визуально + в фильтре
  «Открытые»). Считать ППР только по нарядам, у которых была плановая дата, а не по
  всем открытым (убрать зомби из знаменателя).
- **B.** Разрешить массовое закрытие/отмену старых открытых нарядов из UI (гигиена).
- **C.** И то и другое: A (сигнал) + B (инструмент уборки).

**Критерий готовности (для A):** наряд с `createdAt` > 14 дней назад и статусом
не-DONE/не-CANCELLED показывается с бейджем «просрочен»; ППР считает только наряды с
плановой датой.

**Оценка:** A — ~1 день (логика возраста + правка формулы ППР + фильтр). B — ~день.
