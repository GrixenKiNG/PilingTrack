# Дизайн: единый модуль ТО (объединение + сборка чек-листов из блоков)

Дата: 2026-06-06
Статус: утверждён пользователем (Q&A + макет), к реализации.
Ветка: `feat/inspections-checklist-engine` (продолжение).

## 1. Проблема

Сейчас три отдельных раздела дублируют один смысл «обслуживание техники»:
- **«Обслуживание»** — `MaintenanceRecord` (наряды: ремонт / неисправность / плановое).
- **«Осмотры»** — `Inspection` (прохождение чек-листа).
- **«Чек-листы»** — `ChecklistTemplate` (шаблоны).

Чек-листы привязаны к технике плоской строкой `appliesToModel` и не учитывают
состав машины (молот гидро/дизель, наличие вращателя).

## 2. Решения (утверждены)

1. **Модель техники — одна запись + атрибуты.** Установка остаётся одной записью
   `Equipment`; добавляются атрибуты, по которым подбирается чек-лист.
2. **Чек-лист собирается из блоков** по трём признакам: марка/модель (регламент
   производителя) + тип молота + наличие вращателя.
3. **Единый журнал ТО** — одна сущность «запись ТО» с типом
   (ЕО / ТО-1 / ТО-2 / ТО-3 / сезонное / ремонт / неисправность). Для ЕО/ТО —
   собранный чек-лист; для ремонт/неисправность — свободный ввод (две стадии).
4. **ЕО = осмотр И обслуживание.** В движке это уже выражается: пункты
   обслуживания — тип `DONE` («Выполнено»), проверки — `YES_NO` / `MEASURE`.
   Структурных изменений не требует; вопрос наполнения блоков из регламентов.

## 3. Модель данных

### 3.1 Equipment (+2 поля)
```
hammerKind  HammerKind  @default(NONE)   // HYDRAULIC | DIESEL | NONE
isCombined  Boolean     @default(false)  // есть вращатель
```
`enum HammerKind { HYDRAULIC DIESEL NONE }`. Редактируются в карточке техники.
Существующее поле `kind` (EquipmentKind) и `model` остаются.

### 3.2 ChecklistTemplate — блоки (+2 поля)
```
blockType          BlockType   @default(BASE)  // BASE | HAMMER | ROTARY
appliesToHammerKind HammerKind?                // только для HAMMER-блоков
// appliesToModel (есть) — для BASE-блоков: марка/модель
```
`enum BlockType { BASE HAMMER ROTARY }`.

Подбор блоков для (equipment, level):
- **BASE**: `blockType=BASE AND level=L AND appliesToModel = equipment.model` (или
  `appliesToModel IS NULL` как общий запас — fallback, не «все подряд»).
- **HAMMER**: `blockType=HAMMER AND level=L AND appliesToHammerKind = equipment.hammerKind`
  (пропускается, если `hammerKind=NONE`).
- **ROTARY**: `blockType=ROTARY AND level=L` — только если `equipment.isCombined=true`.

Все запросы tenant-scoped строгим равенством `tenantId` (fail-closed; без
`IS NULL OR` — см. CLAUDE.md правило IDOR).

### 3.3 MaintenanceRecord = «запись ТО» (журнал)
Переиспользуем существующую таблицу (сохраняет данные и UI нарядов P1b).
- `MaintenanceType` расширяется: `EO TO1 TO2 TO3 SEASONAL` (к существующим
  `REPAIR FAULT`). Существующие `SCHEDULED`/`INSPECTION` маппятся: `SCHEDULED→TO1`
  (данных мало; миграция данных в .sql), `INSPECTION→EO`. Значения enum не удаляем
  в этой миграции (недеструктивно), помечаем deprecated в комментарии.
- Две стадии для ремонт/неисправность: стадия 1 — диагностика (`faultCause` +
  фото), стадия 2 — выполненные работы (новое поле `workDone` text + `partsUsedText`
  + фото). Фото через Media (`entityType='maintenance'`).
- `acceptedById`/`acceptedAt` (новые) — «Принять» админом (поверх существующего
  `closedById`). Допускается переиспользовать `closedById`/статус `DONE`; решаем
  на этапе плана.

### 3.4 Inspection — привязка к записи ТО
```
maintenanceRecordId String? @unique   // 1:1 с записью ТО типа ЕО/ТО
```
Для ЕО/ТО: запись ТО создаётся вместе с Inspection (1:1). `templateSnapshot`
теперь хранит **склеенные секции нескольких блоков** (BASE+HAMMER+ROTARY) —
формат снимка уже секционный, расширяется без слома.

## 4. Сборка чек-листа (доменная логика)

Новая чистая функция в `src/modules/inspections/domain/`:
```
composeChecklist(blocks: TemplateBlock[], equipment, level): ComposedSnapshot
```
- На вход — уже отобранные блоки (выборка из БД — в application-слое).
- Порядок секций: BASE → HAMMER → ROTARY; внутри — `section.order`, `item.order`.
- Возвращает плоский снимок секций+пунктов (как сейчас `templateSnapshot`).
- Покрыто unit-тестами: NONE-молот пропускает HAMMER; не комбинированная
  пропускает ROTARY; отсутствие BASE → понятная ошибка (нельзя начать осмотр).

Выборка блоков (application, tenant-scoped) — отдельная функция, тестируется
на 404/кросс-тенант как существующий `startInspection`.

## 5. UI / навигация

- Один пункт меню **«ТО»** вместо «Обслуживание» + «Осмотры». Шаблоны-блоки —
  вкладка для админа (или /admin/checklists, остаётся).
- Страница модуля (см. макет `mockups/to-module-mockup.html`):
  - выбор машины + карточка атрибутов (тип машины · молот · комбинированная);
  - вкладки: **ТО и осмотры** / **Ремонт и неисправности** / **Шаблоны-блоки**;
  - запуск осмотра показывает, **какие блоки соберутся** (предпросмотр состава);
  - история — единый журнал по технике (ЕО/ТО/ремонт вперемешку, Health Score,
    статус Принят/На приёмке/Закрыт).
- Заполнение осмотра — существующий `run-inspection`, секции сгруппированы по
  блокам (BASE/HAMMER/ROTARY) с цветовыми подзаголовками.

## 6. Срезы реализации

- **R1 — модель + сборка (бэкенд):** поля Equipment, поля ChecklistTemplate,
  enum-ы, `Inspection.maintenanceRecordId`, расширение `MaintenanceType`;
  недеструктивная миграция + RLS для новых столбцов; `composeChecklist` +
  выборка блоков; старт ЕО/ТО создаёт запись ТО + Inspection из блоков. TDD.
- **R2 — единый журнал (UI):** страница ТО, карточка атрибутов, вкладки,
  объединённая история по технике, предпросмотр состава блоков, редактирование
  `hammerKind`/`isCombined` в карточке техники.
- **R3 — ремонт/неисправность две стадии + приёмка:** свободный ввод в журнале,
  фото по стадиям, «Принять» админом, разделение истории ТО/Ремонт.

Роли (машинист видит только свою технику; механик/админ) — отдельный последующий
срез, в этот редизайн не входит (используем текущий `maintenance.manage`).

## 7. Границы (YAGNI)

- Нет генерации PDF (нигде).
- Нет автоматики: счётчики наработки, предиктив, автопланирование — «§10».
- Нет видео.
- Голосовой ввод — позже (мобильная диктовка / Web Speech API), не в этих срезах.
- Не строим параллельный модуль — расширяем существующие `inspections` +
  `MaintenanceRecord`.

## 8. Безопасность / совместимость

- Все новые таблицы/запросы tenant-scoped строгим равенством, fail-closed.
- Миграция недеструктивная (только ADD COLUMN / ADD ENUM VALUE) — безопасна для
  локальной копии прод-данных; БД не сбрасываем.
- Новые столбцы наследуют RLS политики (audit-mode) как соседние.
- На проде: новая миграция ⇒ пересобрать `migrate` сервис (см. CLAUDE.md).
