# Модуль «Обслуживание» — конспект переделки (ЧЕРНОВИК)

> Статус: **обсуждение**, не утверждено. Накапливается по ходу разговора, будет корректироваться.
> Дата начала: 2026-06-03. Сменяет/уточняет ранее реализованный P1b (наряды UI).

## 1. Суть наряда — две стадии

Наряд живёт в две симметричные стадии, у каждой один набор вложений (**текст + фото + документы**):

- **Неисправность / диагностика (до):**
  - Причина неисправности — текст (можно надиктовать голосом → расшифровка)
  - **Фото-фиксация поломки** — быстрые фото с телефона
  - **Отчёт диагностики** — документ: PDF и/или фото (от сторонних инженеров: снятие ошибок с блоков, диагностика; иногда отчёт = сфотканные листы)
- **Выполненные работы (после):**
  - Что сделано — текст (голосом)
  - **Фото-фиксация работ/замены** — быстрые фото результата
  - **Отчёт о проделанной работе** — документ: PDF и/или фото
  - + запчасти, трудочасы, стоимость (уже есть)

> Итого в каждой стадии **два вида вложений**: (1) фото-фиксация (оперативные фото) и (2) отчёт-документ (PDF и/или фото-страницы). Текст с голосом — отдельно.

## 2. Голосовой ввод

- Это **расшифровка речи в текст** (надиктовал → попало в поле), НЕ аудиофайл.
- Везде, где есть текст (причина, что сделано).
- **Телефоны (iOS+Android):** встроенный микрофон экранной клавиатуры (ноль кода, работает везде).
- **Десктоп-браузер:** наша кнопка-микрофон (Web Speech API, Chrome/Edge).
- Серверную расшифровку аудио НЕ делаем (дорого, грузит VPS, не нужно).

## 3. Вложения

- **Фото** делятся на «до ремонта» и «после ремонта» (не общая куча).
- Два вида вложений в каждой стадии:
  1. **Фото-фиксация** — изображения, оперативные фото (поломка / результат работ).
  2. **Отчёт** — документ: **PDF и/или фото** (диагностика инженеров; отчёт о работах). Word/Excel/прочее НЕ нужны (убрали).
- Потолки: PDF — 25 МБ, фото — 10 МБ.
- Прикреплять можно **уже при создании** наряда (приём: id наряда генерируется заранее, как у сменных отчётов).
- Бэкенд-хранилище уже принимает любые форматы — добавим только метку «к причине» / «к работе».

## 4. Роли и процесс

- Ведут наряды **оператор и механик**.
- Любой уполномоченный ведёт наряд целиком: **сам открывает, ремонтирует, закрывает**. Жёсткой передачи «оператор завёл → механик закрыл» НЕ требуется.
- **Админ** может всё сам + управляет правами и при необходимости заводит доп. роли (напр. отдельная роль «механик»).

**Видимость по ролям:**
- **Оператор** — видит **только свои** наряды.
- **Механик** — видит **все** наряды.
- **Админ / диспетчер** — видят все.

**Приёмка:**
- После статуса «Выполнено» работу принимает **только админ** — кнопкой **«Принять»** → статус **«Принят»** (финальное подтверждение).
- Статусы: Новая → (Назначена) → В работе → (Приостановлена) → Выполнено → **Принят**. (Отменена — отдельно.)

**Поиск/фильтр:**
- По нарядам — **поиск по установке** (фильтр по машине). Доп. фильтры (тип, статус) уже есть.

## 5. Платформы

- Работают с **Android, iOS и десктопного браузера**.
- ⚠️ **Риск (см. §7):** браузерное распознавание речи на iOS не работает штатно.

## 6. Объём (что НЕ делаем сейчас)

- Без автоматики: нет напоминаний о ТО по моточасам/датам, нет показателей (MTBF/MTTR), нет уведомлений. Отложено.
- Не дублировать: вкладку ТО в карточке установки свести к ссылке на наряды этой машины (убрать второе место с тем же самым).
- **Печать/экспорт наряда в PDF — НЕ нужно** (отдавать заказчику/в архив не требуется).

## 6a. История по установке (раздельно ТО / Ремонт)

- На карточке установки — **история обслуживания именно этой машины**.
- Раздельно по типу: **ТО** (плановое) и **Ремонт** (и неисправность/осмотр). Фильтр/группировка по типу.
- Это основной «журнал машины»; доска `/admin/maintenance` остаётся общим списком по всем машинам.

## 6b. Чек-листы ТО (по регламенту / тех. документации)

- На каждое ТО — **чек-лист** по регламенту тех. документации и руководящим документам.
- **Шаблон чек-листа** = именованный набор пунктов (что проверить/сделать), заводит админ из регламента.
- **Привязка шаблонов: вариант В** — админ ведёт **библиотеку** именованных шаблонов; при плановом ТО механик выбирает нужный шаблон из списка. Авто-подсказку по модели добавим позже (когда «модель» станет справочником).
- **Уровни ТО: ТО-1, ТО-2, по моточасам.** Каждый уровень — свой шаблон (напр. «ТО-1 Banut 655», «ТО по моточасам 250»). Поле «уровень ТО» у шаблона.
  - «по моточасам» здесь = категория ТО (заполняется вручную), НЕ авто-триггер по наработке (автоматика отложена, см. §6).
- При выполнении планового ТО механик/оператор **заполняет чек-лист**: отмечает пункты (выполнено / не выполнено / примечание).
- Заполненный чек-лист хранится в наряде (часть «Выполненных работ»), виден в истории.
- Чек-лист — только для типа **ТО**. Ремонт/неисправность ведутся свободным текстом (причина/работы), без чек-листа.
- **Пункт чек-листа (вариант 2):** галочка (выполнено / не выполнено / не требуется) + **необязательное значение/замер** (число с единицей или короткий текст) + примечание. Админ при создании шаблона решает, у каких пунктов нужен замер.

4. ✅ Привязка шаблонов чек-листов — **вариант В** (библиотека + ручной выбор при ТО).
5. ✅ Уровни ТО — **ТО-1, ТО-2, по моточасам**.
6. ✅ Пункт чек-листа — **вариант 2** (галочка + необязательный замер + примечание).

## 8. Разбивка на части (предложение)

Набралось на 3 независимых куска. Каждый = свой спец → план → реализация.

- **Ч1. Двухстадийный наряд + документы.** Переделать карточку/диалог наряда на два блока «Неисправность/диагностика» и «Выполненные работы», в каждом текст + фото + документы. Прикрепление при создании (id заранее). Голос-кнопка на десктопе; на телефоне — родная диктовка. Разделение фото/доков по меткам «до»/«после».
- **Ч2. История по установке.** Журнал обслуживания на карточке машины, раздельно ТО / Ремонт. Доска остаётся общим списком.
- **Ч3. Чек-листы ТО.** Админ-библиотека шаблонов (уровни ТО-1/ТО-2/по моточасам), пункты «галочка + необязательный замер + примечание». Заполнение чек-листа при ТО, хранение в наряде, видно в истории.

- **Ч4. Справочник расходников по установке.** На карточке машины (паспорт) — список расходных материалов с нормами. Поля: **наименование + марка/размер + норма (кол-во) + единица + примечание** (пока достаточно, при необходимости расширим, напр. артикул). У каждой машины свой (свайный канат, молотовой канат и т.д.). Цель: открыл — видишь, что и сколько заказать, не роясь в записях. **Статический справочник норм, НЕ складской учёт остатков** (живой склад — отдельная большая тема, отложено).

**Порядок:** ядро — **Ч1 → Ч2 → Ч3** (подтверждено). **Ч4** независимый и маленький — можно вставить первым как быстрый полезный или сделать параллельно.

## 10. БОЛЬШАЯ ВИЗИЯ — целевая архитектура эксплуатации (V2+)

> Добавлено 2026-06-03. Это «север», а не ближайший спринт. **Разворачивает ранние решения** «без автоматики» и «без узлов» (см. §11 — расхождения). Часть формулировок из источника привязана к Supabase (Edge Functions, `rpc()`, RLS) — у нас стек **Next.js + Prisma + Postgres + outbox/projection-воркеры + cron**, переносим понятия на него.

### 10.1 Производство → счётчики наработки (event-driven)
- **Парк смешанный:** копры, дизель-молоты, гидромолоты.
- **Сменный журнал = источник наработки.** Уже фиксирует фактическую работу: связка (копёр + молот), сваи, удары, часы вибрации, моточасы, расход топлива. Те же цифры переиспользуем для ТО — не вводим вручную.
- **Событие `production.session.closed`** при закрытии смены/записи по свае → по **маппингу типа/модели техники** превращается в показания. Одна смена кормит несколько машин: копёр +моточасы; молот +удары +часы вибрации.
- **Две таблицы:** `meter_readings` — журнал дельт только на добавление (аудит, прогноз, история отказов); `meters` — агрегат `current_value` на пару (техника + тип счётчика).
- **Идемпотентность:** `unique(meter_id, session_id)`, агрегат растёт только если строка реально вставилась (ретраи/переоткрытие смены безопасны).
- **Три источника без двойного счёта:** инкременты (удары/сваи/вибро) — из производства; **моточасы** — приоритет телематики (абсолют), иначе из производства; **мост для неинструментованной техники** — нет счётчика ударов → оценка `сваи × средние удары/сваю` (коэф. по технике того же типа).

### 10.2 Регламенты и прогноз
- **Типы:** ЕО / ТО-1 / ТО-2 / СО / ТР / КР + предиктив + аварийные наряды.
- Таблицы регламентов: `maintenance_plans / plan_intervals / plan_tasks / plan_parts`.
- **Пересчёт `next_due`** на каждое показание; правило **«что раньше»** (счётчик ИЛИ календарь). Формулировка пользователю: не «просрочено», а **«через 4 смены подойдёт ТО-1»**.
- При приближении/переходе порога — **авто-создание наряда + резерв ЗИП**.
- Наряд переводит машину в ТО → машина **авто-недоступна для планирования работ**.
- Дёшево на закрытии смены (пересчёт по одной машине в транзакции, Postgres-функция); тяжёлый прогноз по парку — по cron (воркер).

### 10.3 Единый движок чек-листов/инспекций
- **Объединить «Инспекции» и «ЕО-чек-листы» в ОДИН конфигурируемый движок** — не плодить дубль (это и есть Ч3 в расширенном виде).
- Структура: шаблоны → условия → пункты → инстансы → медиа.
- **Динамика:** у пункта поле `condition` (null = всегда); `applicable_items()` сверяет условия с `meters` + сезон (напр. базовые пункты + наработка>2000 мч + зима).
- **Доказательная база:** `start_inspection()` пишет `checklist_snapshot` (какие пункты показаны + наработка на момент) — даже если шаблон потом изменят.
- **Серверная гарантия:** `complete_inspection()` на стороне БД проверяет незаполненные обязательные пункты и пункты без обязательного фото → при пропусках бросает ошибку, статус `completed` не ставится. Клиент не обойдёт.
- **Данные о состоянии вместо телеметрии:** замеры с `feeds_param` (давление/темп/компрессия) → `condition_readings`. Поскольку сенсоров почти нет — это **единственный канал данных о состоянии**, основа предиктива.
- **ИИ-советник:** `ai_findings` на медиа (флаги-рекомендации), решение человека в `ai_confirmed`; авто-закрытие пунктов нейросетью отключено.

### 10.4 Аналитика операторов и KPI
- Поведение операторов из производственных данных (напр. «оператор №14 даёт +31% перегрузок»). Редкая и ценная функция.
- **Целевые KPI (обслуживание по состоянию, из пакета 50PR):** коэффициент готовности установки · наработка до отказа (MTBF) · MTTR · дефектов на 100 м/ч · ресурс канатов · ресурс РВД · ресурс гидромолота · стоимость эксплуатации на 1 сваю.
- **Обязательная фотофиксация** (уровень гидромасла, канаты, молот, мачта, РВД, ходовая, площадка) — сырьё для аналитики и доказательная база. Всё это — §10-«север» (переход от планового ТО к обслуживанию по состоянию), НЕ ближняя ручная цель.

### 10.5 ER-модель (mermaid — позже)
- техника → узлы → счётчики → регламенты → наряды → ЗИП.

### 10.6 Замыкание цикла (два входа → один наряд)
- **Плановый путь:** производство пишет наработку в счётчики → `recompute_due()` (полная версия, заменяет заглушку) считает **first-to-reach** по счётчикам и календарю → при переходе порога создаёт **черновик планового наряда** + резерв ЗИП через `reserve_parts_for_wo()`.
- **Путь от осмотра:** замер вне допуска (`passed=false`) ИЛИ подтверждённый AI-флаг (`ai_confirmed=true`) → триггер `tg_answer_defect` заводит **дефект** → подтверждение механиком → `create_wo_from_defect()` создаёт **корректирующий наряд** с приоритетом по серьёзности. («осмотр → дефект → наряд».)
- **Схождение:** оба пути → `work_orders`, переходы ведёт `transition_work_order()` — валидирует по конечному автомату (диаграмма статусов), пишет неизменяемый аудит в `wo_events`; **при закрытии** снимает резерв ЗИП и фиксирует `meter_at_service`.
- **Сброс цикла:** снимок наработки `meter_at_service` — точка отсчёта; `recompute_due()` начинает считать до следующего ТО заново.

## 11. Расхождения визии (§10) с ранними решениями

1. ✅ **«Без автоматики»** — остаётся для Ч1–Ч4. Предиктив/авто-наряды (§10) — будущее.
2. ✅ **«Без узлов»** — остаётся; наряд цепляется к машине целиком. Узлы (§10.5) — будущее.
3. **Стек.** Источник написан под Supabase (Edge Functions/`rpc()`/RLS). У нас Next.js+Prisma+Postgres+outbox/воркеры+cron — переносим 1:1 по смыслу, без Supabase.
4. **Зависимость от Сменного журнала.** Нужно проверить, что он реально пишет по каждой единице связки: удары, часы вибрации, моточасы (не только сваи). Если нет — сначала доработать производственный контур.
5. **Телематика дремлет** (нет железа). Значит «абсолют от телематики» — будущее; живой путь сейчас = производство + мост-оценка. Хорошо, что дизайн это уже учитывает.
6. **Масштаб = платформа, не спринт.** Ч1–Ч4 остаются верным фундаментом; §10 — пункт назначения. Нужна дорожная карта фаз.

## 12. ЕО — ежедневный осмотр (детальная заявка 2026-06-03)

> По сути это **конкретная реализация Ч3 (движка чек-листов)** для ежедневного предсменного/послесменного осмотра. ВАЖНО: ЕО-осмотр — отдельная сущность от ремонтного наряда; осмотр может **породить** ремонтный наряд при дефекте (ср. §10.6 «осмотр → дефект → наряд»).

**Создание чек-листа:** выбор техника / дата / смена / оператор / механик / объект; авто-подтяг: серийный №, наработка (моточасы), последнее ТО, статус, активные аварии/предупреждения.

**10 разделов осмотра:** 1) Внешний осмотр, 2) Двигатель, 3) Гидросистема, 4) Лебёдки и канаты, 5) Мачта и рабочее оборудование, 6) Ходовая часть, 7) Электрооборудование, 8) Приборы безопасности, 9) Телеметрия (авто), 10) Пробный запуск. (Полный список пунктов — в исходной заявке.)

**Типы ответа на пункт:** Исправно / Замечание / Неисправно / Не проверено.

## 18. ДИЗАЙН движка чек-листов (Ч3) — УТВЕРЖДАЕТСЯ

> Брейншторм-дизайн, по частям с одобрением пользователя (2026-06-03).

**Подход А (✅ утверждён):** осмотр — отдельная сущность, связанная с нарядом. Осмотр по дефекту порождает ремонтный наряд (Ч1). Не смешиваем осмотр и ремонт.

**Три сущности (✅ утверждены):**
1. **Шаблон чек-листа** — библиотека (админ); привязка к типу/модели техники; уровень (ЕО / ТО-1 / ТО-2 / ТО-3 / сезонное); разделы → пункты.
2. **Пункт шаблона** — текст; тип ответа (Да/Нет · 4-статуса Исправно/Замечание/Неисправно/Не проверено · Выполнено · замер[значение+единица+норма]); норма (может зависеть от модели); фото (обяз./нет); примечание; голос-в-текст; провенанс (стр. руководства); условие показа (сезон/наработка — на будущее, выключено).
3. **Проведённый осмотр** — техника/дата/смена/исполнитель/наработка; ответы+фото/голос/примечания; Health Score (исправные/всего); снимок шаблона (доказательная база); подпись; статус; может породить ремонтный наряд.

**Экраны (✅ утверждены):**
- Админ: «Шаблоны» — список + редактор (разделы/пункты/типы/нормы/провенанс) + дублирование.
- Машинист: «Провести осмотр» — выбор техники → шаблон по модели → форма (фото/голос/примечание, живой Health Score) → серверная проверка обязательных пунктов/фото → подпись → по дефектам предложить ремонтный наряд.
- История осмотров (фильтр техника/дата/уровень); на карточке установки — её осмотры (Ч2).
- Навигация: в «Обслуживание» добавить «Осмотры» и (админ) «Шаблоны».

**Видимость (✅ уточнено):** машинист в «Провести осмотр» видит **только закреплённую за ним технику** (через экипаж/бригаду — существующая связь техника↔оператор, как в `listAllEquipment(operatorUserId)`). Админ/механик — всю.
