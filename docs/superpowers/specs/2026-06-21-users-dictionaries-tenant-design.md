# Пользователи и справочники: tenant-изоляция и операционное усиление

Дата: 21.06.2026
Статус: согласовано к планированию
Область: `/admin/users`, `/admin/dictionaries`, API, Prisma, аудит и кэш

## 1. Цель

Сделать модули «Пользователи» и «Справочники» безопасными для нескольких организаций и полезными в ежедневной работе PilingTrack.

Ключевые решения:

- обычный `ADMIN` управляет только данными своей организации;
- роль `SUPER_ADMIN` на этом этапе не вводится;
- каждая организация владеет собственными справочниками;
- системные шаблоны используются только для первоначального заполнения;
- использованные справочные значения сохраняют исторический смысл отчётов;
- пользователи отображаются вместе с их реальными рабочими назначениями и активностью.

## 2. Границы этапа

Входит:

- tenant-scoped CRUD пользователей;
- tenant-scoped справочники свай, бурения и простоев;
- копирование системных шаблонов при создании организации;
- безопасная миграция существующих справочников;
- запрет изменения использованных значений;
- аудит и инвалидация кэша;
- расширение таблицы и карточки пользователя;
- unit, API, integration и browser E2E тесты.

Не входит:

- `SUPER_ADMIN`;
- произвольный конструктор ролей и разрешений;
- наследование последующих изменений системных шаблонов;
- общий межtenant каталог;
- массовый импорт пользователей и справочников.

## 3. Модель доступа

### 3.1 Пользователи

- `ADMIN` получает доступ только к пользователям с тем же `tenantId`.
- List, read, update, block, archive и delete обязаны использовать условия `id + tenantId`.
- Cross-tenant обращение возвращает `404`, чтобы не раскрывать существование записи.
- Создаваемый пользователь всегда наследует `tenantId` текущего администратора.
- Передача tenant через request header, query или body не может изменить область операции.
- Администратор не может удалить или заблокировать собственную запись.

### 3.2 Справочники

- `PileGrade`, `DrillingType` и `DowntimeReason` принадлежат одной организации.
- Любое чтение и изменение ограничивается tenant текущего пользователя.
- Оператор получает только активные значения своей организации.
- Администратор видит активные, архивные и все значения своей организации.

## 4. Модель данных

### 4.1 Пользователь

Существующие связи используются как источник рабочей информации:

- `UserSiteAssignment` — закреплённые объекты;
- активная `Crew` — бригада и установка;
- `Report` — последний отчёт и производственная активность;
- audit events — история административных изменений.

Для оперативной карточки API возвращает:

- основные данные пользователя;
- список закреплённых объектов;
- активную бригаду и установку;
- дату последнего отчёта;
- количество отчётов;
- дату последней подтверждённой активности.

Последняя активность на первом этапе определяется по максимуму из доступных серверных событий: входа, отчёта и административного обновления. Если входы пока не сохраняются как отдельное поле, API честно возвращает последний известный источник активности, не подменяя его текущим временем.

### 4.2 Справочники организации

Каждая из трёх моделей получает обязательный `tenantId` и связь с `Tenant`.

Уникальность задаётся внутри организации по нормализованному имени. Нормализация включает trim, приведение регистра для проверки дубликата и схлопывание повторных пробелов. Исходное отображаемое имя сохраняется.

Для `PileGrade` добавляются структурированные поля:

- `code` — обозначение марки;
- `lengthMeters` — длина сваи в метрах;
- `sectionOrDiameter` — сечение или диаметр;
- `notes` — примечание.

Расчёт метров должен использовать `lengthMeters`, а не извлечение цифр из отображаемого имени.

### 4.3 Системные шаблоны

Системные шаблоны не участвуют напрямую в рабочих отчётах. Они хранятся отдельно от tenant-owned записей или представлены неизменяемым seed-каталогом.

При создании организации сервис инициализации:

1. читает активные системные шаблоны;
2. создаёт независимые tenant-owned копии;
3. выполняет операцию транзакционно;
4. записывает событие инициализации в аудит;
5. допускает повторный запуск без создания дубликатов.

Последующие изменения системных шаблонов не изменяют данные существующих организаций.

## 5. Историческая неизменяемость

Справочное значение считается использованным, если на него ссылается хотя бы один отчёт или план.

Правила:

- использованное значение можно архивировать и восстановить;
- использованное значение нельзя переименовать или удалить;
- для исправления создаётся новое значение, старое архивируется;
- неиспользованное значение можно переименовать или удалить;
- API возвращает `409` и количество связанных отчётов/планов;
- интерфейс предлагает действие «Архивировать и создать новое».

Архивирование не изменяет уже созданные отчёты, PDF, историю и аналитику. Старые связи продолжают разрешаться, но архивное значение не предлагается в формах новых отчётов.

## 6. Модуль «Пользователи»

### 6.1 Таблица

Колонки:

- ФИО и телефон;
- роль;
- текущий объект;
- бригада и установка;
- последняя активность;
- статус доступа.

Поиск выполняется по ФИО, email и телефону.

Быстрые фильтры:

- все;
- операторы;
- помощники;
- диспетчеры;
- администраторы;
- заблокированные;
- без объекта;
- без бригады;
- неактивные более 30 дней.

KPI:

- всего;
- активные;
- операторы;
- без назначения;
- заблокированные.

### 6.2 Правая панель

Вкладки:

1. **Обзор** — ФИО, телефон, email, роль, статус, дата создания, последний вход или последний известный источник активности.
2. **Закрепление** — объекты, активная бригада, установка, текущая смена при наличии данных.
3. **Активность** — последний отчёт, количество отчётов, дата последней работы.
4. **Доступ** — смена роли, сброс пароля/PIN, блокировка, отзыв сессий.
5. **История** — кто, когда и что изменил.

### 6.3 Жизненный цикл

- Работавший пользователь не удаляется физически: используется блокировка/архивирование.
- Hard delete доступен только для ошибочно созданной записи без отчётов, назначений, бригад и audit-зависимостей.
- Блокировка и смена пароля/PIN отзывают действующие сессии пользователя.
- Пароль содержит минимум 8 символов.
- PIN содержит от 4 до 10 цифр.

## 7. Модуль «Справочники»

Сохраняются три вкладки:

- сваи;
- бурение;
- простои.

Таблица содержит:

- название или обозначение;
- структурированные характеристики, где применимо;
- статус;
- число отчётов;
- число планов;
- дату изменения;
- автора последнего изменения;
- доступные действия.

Сохраняются поиск и фильтры `Активные / Архив / Все`.

Действия:

- создать;
- переименовать неиспользуемое;
- архивировать;
- восстановить;
- удалить неиспользуемое;
- архивировать использованное и открыть форму создания нового значения на его основе.

## 8. API и сервисы

### 8.1 Пользователи

Основной users query получает tenant из authenticated session и передаёт его в service-layer. Service-layer не принимает tenant из пользовательского body.

Mutation API:

- валидирует вход через Zod;
- проверяет `users.manage`;
- выполняет tenant-scoped lookup до изменения;
- записывает аудит;
- отзывает сессии при security-sensitive изменениях;
- возвращает нормализованный DTO.

### 8.2 Справочники

Dictionary query и mutations принимают tenant только из authenticated session.

После успешной mutation выполняются:

1. запись audit event;
2. инвалидация data cache tenant;
3. инвалидация response cache tenant;
4. возврат изменённой записи или результата операции.

Ошибки:

- `400` — неверные данные;
- `404` — запись отсутствует в tenant текущего пользователя;
- `409` — дубликат или запрещённое изменение использованной записи;
- `403` — недостаточно роли/разрешения.

## 9. Миграция существующих данных

Миграция выполняется транзакционно и повторно запускаемо.

Последовательность:

1. добавить nullable tenant/структурированные поля;
2. создать tenant-копии существующих глобальных значений;
3. построить отображение `старый id -> tenant-копия id`;
4. переключить `PileWork`, `LeaderDrilling`, `ReportDowntime` и `SitePilePlan` по tenant их родительского отчёта/объекта;
5. заполнить `lengthMeters` из существующих планов или однозначно распознаваемого обозначения;
6. неоднозначные значения вывести в отдельный migration report и не подменять нулём;
7. сверить количество связей до и после;
8. сделать tenant обязательным и добавить уникальные индексы;
9. удалить глобальные записи только после успешной проверки отсутствия ссылок.

Миграция должна завершаться ошибкой при неоднозначном tenant, потерянной связи или несовпадении контрольных количеств.

## 10. Состояния и ошибки интерфейса

Оба модуля обязаны поддерживать:

- loading skeleton;
- пустой список;
- отсутствие результатов фильтра;
- отдельное состояние ошибки;
- кнопку повторной загрузки;
- disabled/submitting состояния mutation;
- понятный текст для `409` с предлагаемым следующим действием.

Клиентское состояние изменяется только после успешного ответа API. При ошибке оптимистическое состояние не сохраняется.

## 11. Тестовая стратегия

### Unit

- нормализация названий;
- проверка использования справочного значения;
- правила rename/archive/delete;
- расчёт фильтров и KPI пользователей;
- формирование activity/assignment DTO.

### API

- ADMIN tenant A не читает и не изменяет пользователей tenant B;
- ADMIN tenant A не читает и не изменяет справочники tenant B;
- tenant нельзя подменить через header/query/body;
- проверка ролей, validation и error mapping;
- блокировка и credential reset отзывают сессии;
- mutation справочника пишет аудит и сбрасывает кэш.

### Integration с реальным PostgreSQL

- миграция существующих данных;
- уникальность внутри tenant и допустимое совпадение между tenant;
- сохранение всех FK после remap;
- запрет cross-tenant mutation;
- повторный запуск инициализации шаблонов;
- неизменность старых отчётов после архивирования справочника.

### Browser E2E

- поиск и фильтры пользователей;
- правая панель и вкладки;
- блокировка пользователя;
- создание, архивирование и восстановление справочника;
- отказ переименования использованной записи;
- desktop и mobile размеры;
- loading, empty и error состояния.

## 12. Критерии приёмки

- Ни один users/dictionary запрос не возвращает данные другой организации.
- `ADMIN` не может выбрать или передать чужой tenant.
- Каждая новая организация получает независимый набор шаблонов.
- Совпадающие названия разрешены в разных организациях и запрещены внутри одной.
- Использованное значение не переименовывается и не удаляется.
- Архивирование не меняет старые отчёты, PDF, историю и KPI.
- Метры свай считаются по структурированной длине.
- Пользовательская карточка показывает только реальные назначения и активность.
- Все изменения пользователей и справочников видны в аудите.
- Focused tests, build и browser E2E проходят без регрессий.
